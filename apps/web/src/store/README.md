# store

Estado global con Zustand.

| Archivo              | Uso                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **workshopStore.ts** | Todas las llamadas al backend usan **`apiFetch`** (`utils/apiClient`) con JWT. Exporta **`sessionMessageBody`** para `POST /sessions/:id/messages` con `stageId` opcional. Store del Workshop: project (incluye **`stages[]`**, **`complexityPending`** HITL), **`workshopStages`** (copia sincronizada de `stages[]` para el selector multi-etapa), **`activeStageId`** (vista en vivo MDD), session, mddContent, loading, synced, error, **agentProgress**, **evaluatorCritique**. **`conformance.blueprintDataModel`**: alineación Blueprint vs **§3 modelo de datos** del MDD; si `ok === false`, **`generateApiContracts`** no llama al API (y el backend también devuelve 400). **`confirmComplexityProposal` / `dismissComplexityProposal`** para el banner de complejidad. Tras stream DBGA (`/ai-analysis/stream`), si el `done` incluye `complexityProposal`, se actualiza el proyecto en memoria antes de `fetchProject`. **`clearChat`** borra también `evaluatorCritique` tras un borrado de historial exitoso. **`setActiveStageId`** usa `workshopStages` o `project.stages`, refresca `managerThreadId` vía `GET /ai-analysis/mdd/thread?stageId=` y **`fetchEstimation`**; **`createWorkshopStage`** (POST `/projects/:id/stages`) interpreta **`{ stage }`**, fusiona en `workshopStages` y foco en la nueva etapa (fallback `fetchProject` si el formato no es el esperado). **`fetchWelcome`** envía `stageId`; **`persistMddContent`** y **`fetchEstimation`** envían `stageId` cuando hay etapa activa; **`sendMessage`** incluye `stageId` en el body del orquestador; **MDD stream** incluyen `stageId` cuando aplica. **Conformance** se actualiza al cargar proyecto (`fetchConformance`) y tras generar entregables. **`generateDeliverablesCascade`** → `POST /projects/:id/generate-deliverables`; si el API devuelve `{ queued, jobId }` (BullMQ + `REDIS_URL`), hace **polling** a `GET …/deliverables-jobs/:jobId` y actualiza `agentProgress` hasta `completed`, luego `fetchProject`. |
