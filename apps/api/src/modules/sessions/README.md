# Sessions

Gestión de sesiones de chat del Workshop y parseo de respuestas que incluyen documentos (MDD, UX/UI, DBGA, BRD por etapa, Manual To-Be, etc.).

- **SessionsService:** CRUD de sesiones, `chat`, `chatStream`, `generateWelcome`. `chat` / `chatStream` los usa **`AiOrchestratorService`** (persistencia del log y llamada al modelo); no son el contrato HTTP de `POST :id/chat`. Opciones LLM: `currentBrdContent` / `currentToBeManualContent` cuando `activeTab` es `brd` o `to-be`; el `done` puede devolver `brdContent` / `toBeManualContent` parseados con `---FIN_BRD---` y `---FIN_TOBE---`. **`generateWelcome`:** pasa `activeTab` y **`welcomeBrief: true`** al `AiService` (system corto en lugar de `MASTER_PROMPT` + sin inyectar documentos enteros en system; el contexto acotado va en el **user** `syntheticPrompt`). Ante **429** del proveedor (p. ej. OpenRouter) puede aplicarse reintento con backoff según orquestación; en rate limit sostenido puede persistirse un **mensaje estático** de bienvenida (evita 500 en producción). Mensajes en `chatLog` pueden incluir **`stageId`**. `POST :id/messages` acepta `stageId` opcional (`appendChatSchema`). **Imágenes (hasta 6):** `describeImagesForChat` usa el **modelo de visión** de la instancia activa (`createForVisionUser`); el bloque `VISION_CONTEXT_HEADER` queda en `content` (las `images` en el log son solo UI). El **stream de chat** envía al LLM solo texto (`sessionHistoryToLlm` sin binarios) para no invocar el modelo de chat con `image_url` (p. ej. deepseek → 404).
- **`POST :id/chat` (SessionsController):** delega en **`AiOrchestratorService.chatBySessionId`** — misma lógica que **`POST /ai-orchestrator/chat`** (HITL complejidad, PATCH MDD/UX/DBGA desde body, supervisor, legacy/TheForge, **`uxGuideLlmOptions`** cuando toca, ingest SDD, evaluador). Body: `message` y/o **`images`** (`parseChatImageAttachments`), opcionales `activeTab`, `stageId`, `mddContent`, `uxUiGuideContent`, `dbgaContent`. Respuesta: **`{ session, project, uxUiGuideContent?, evaluatorCritique? }`** (no el retorno plano interno de `SessionsService.chat`).
- **ChatResponseParserService:** Separa documento vs mensaje de chat (delimitadores `---FIN_MDD---`, `---FIN_DBGA---`, etc.), fusiona secciones de MDD/DBGA y fallback cuando el modelo omite el delimitador (incl. Fase 0 / tab `benchmark`, títulos tipo *Research Report*). `SessionsService.salvageDbgaFromAssistantText` recupera DBGA pegado solo en el chat. Si el modelo responde solo «He actualizado…» sin `---FIN_DBGA---`, `refineDbgaFromUserRequest` relanza el refinado con `BENCHMARK_REFINE` y el DBGA actual. Con **imágenes**, la descripción de visión va al refinado (no solo el texto corto del usuario). `benchmarkAssistantChatMessage` evita el falso «Benchmark actualizado» si no se persistió nada. Tab **benchmark** + sub-tab Fase 0 edita `dbgaContent`; el Benchmark (Deep Research) es `phase0SummaryContent` (no se pisan entre sí).
- **document-content.util.ts:** Funciones puras `normalizeDashes`, `stripChatLabel`, `cleanDocumentContent` (sin DI). Usado por el parser y por Projects/Legacy para limpiar contenido antes de persistir.
