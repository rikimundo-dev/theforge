# Sessions

Gestión de sesiones de chat del Workshop y parseo de respuestas que incluyen documentos (MDD, UX/UI, DBGA, BRD por etapa, Manual To-Be, etc.).

- **SessionsService:** CRUD de sesiones, `chat`, `chatStream`, `generateWelcome`. `chat` / `chatStream` los usa **`AiOrchestratorService`** (persistencia del log y llamada al modelo); no son el contrato HTTP de `POST :id/chat`. Opciones LLM: `currentBrdContent` / `currentToBeManualContent` cuando `activeTab` es `brd` o `to-be`; el `done` puede devolver `brdContent` / `toBeManualContent` parseados con `---FIN_BRD---` y `---FIN_TOBE---`. **`generateWelcome`:** pasa `activeTab` al `AiService` (instrucciones y logs alineados al tab); ante **429** de Gemini reintenta hasta 4 veces con backoff exponencial; si sigue en rate limit, persiste un **mensaje estático** de bienvenida (evita 500 en producción). Mensajes en `chatLog` pueden incluir **`stageId`**. `POST :id/messages` acepta `stageId` opcional (`appendChatSchema`).
- **`POST :id/chat` (SessionsController):** delega en **`AiOrchestratorService.chatBySessionId`** — misma lógica que **`POST /ai-orchestrator/chat`** (HITL complejidad, PATCH MDD/UX/DBGA desde body, supervisor, legacy/TheForge, **`uxGuideLlmOptions`** cuando toca, ingest SDD, evaluador). Body: `message` y/o **`images`** (`parseChatImageAttachments`), opcionales `activeTab`, `stageId`, `mddContent`, `uxUiGuideContent`, `dbgaContent`. Respuesta: **`{ session, project, uxUiGuideContent?, evaluatorCritique? }`** (no el retorno plano interno de `SessionsService.chat`).
- **ChatResponseParserService:** Separa documento vs mensaje de chat (delimitadores `---FIN_MDD---`, etc.), fusiona secciones de MDD y limpia contenido. Una sola responsabilidad: parseo de respuesta.
- **document-content.util.ts:** Funciones puras `normalizeDashes`, `stripChatLabel`, `cleanDocumentContent` (sin DI). Usado por el parser y por Projects/Legacy para limpiar contenido antes de persistir.
